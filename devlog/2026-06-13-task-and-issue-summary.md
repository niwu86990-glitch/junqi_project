# 2026-06-13 项目任务总结与问题总结

## 一、今日工作概述

本文件汇总 2026-06-12 至 2026-06-13 这一轮连续开发工作的最终结果。

本轮开发围绕军棋识别系统在 X210 开发板上的实际使用体验展开，主要完成了
四个方向的工作：

1. 将原有“一张图片同时放置两枚棋子”的识别方式，改为红方和黑方分别识别、
   分别确认的双阶段流程。
2. 调整 X210 主界面、识别弹窗和胜负结果的显示方式，使界面更适合
   `1024×600` 开发板屏幕和触摸操作。
3. 在不修改现有字符模板库的前提下，重构棋子检测、字符提取、模板匹配和
   颜色分类流程，提高复杂背景、位置变化、光照变化和红色反光情况下的稳定性。
4. 增加“已隐藏结果再次查看”功能，并完善 ARM 编译、部署打包和 Windows
   桌面自动复制流程。

本轮工作的核心思路不是简单增加固定阈值，而是减少识别算法对拍摄环境的
假设，让同一枚棋子在不同光照、背景和摆放位置下，经过处理后仍能形成较为
一致的字符特征。

## 二、识别交互流程重构

### 2.1 原有流程的问题

原流程要求一张图片中同时出现红、黑两枚棋子。这种方式存在以下问题：

- 两枚棋子必须同时处于合适的位置，摆放要求较高。
- 摄像头视野需要同时覆盖两枚棋子，单枚棋子的有效像素较少。
- 两枚棋子的反光、阴影和背景可能相互影响。
- 识别结果会同时出现，不利于在对战中保护双方棋子信息。
- 任意一方识别错误时，整次拍摄都可能需要重新进行。

### 2.2 新流程

新流程改为双方独立完成识别：

1. 红方将自己的棋子放在摄像头下。
2. 红方点击“红方识别”按钮。
3. 开发板截图并识别棋子颜色、字符和置信度。
4. 弹窗仅向当前操作者展示识别结果。
5. 用户可以选择“确定”或“识别失误”。
6. 选择“识别失误”时，可以从 12 种棋子中人工选择正确结果。
7. 确认后，主界面只显示“结果已隐藏”，不直接显示棋名。
8. 黑方按照同样流程完成识别。
9. 双方都确认后，结果进入 `BattleJudge` 判定流水线。
10. 开发板显示红方获胜、黑方获胜或平局。
11. 点击“开始下一轮”后，清除双方保存的结果并恢复初始状态。

### 2.3 状态管理思路

主窗口使用以下状态控制整轮流程：

- `redPiece_`：保存红方最终确认的识别结果。
- `blackPiece_`：保存黑方最终确认的识别结果。
- `redReady_`：红方是否完成确认。
- `blackReady_`：黑方是否完成确认。
- `activeSide_`：当前是否有一方正在执行截图和识别。

只有 `redReady_` 和 `blackReady_` 同时为真时，程序才会执行对战判定。
这可以避免某一方尚未确认时提前进入判定，也避免识别任务重复触发。

## 三、已隐藏结果再次查看

### 3.1 功能需求

识别结果确认后需要隐藏，防止对手直接看到；但在开始下一轮前，当前用户可能
需要再次核对刚才确认的内容。

### 3.2 实现方式

原来的双方状态区是不可点击的 `QLabel`。本次将它们改为保持相同视觉风格的
`QPushButton`：

- 未识别时显示“等待识别”，按钮禁用。
- 识别过程中显示“正在识别”，按钮禁用。
- 确认后显示“结果已隐藏，点击查看”，按钮启用。
- 点击按钮后，弹窗显示颜色、棋名和置信度。
- 再次查看弹窗只提供“确认”按钮，不允许修改已确认结果。
- 点击确认或关闭弹窗后，主界面仍只显示“结果已隐藏”。
- 对战判定完成后，开始下一轮之前仍然可以查看。
- 点击“开始下一轮”后，保存结果被清空，查看按钮重新禁用。

### 3.3 设计考虑

再次查看与首次确认被设计成两个不同操作：

- 首次确认允许“确定”或“识别失误”，因为此时结果尚未进入最终状态。
- 再次查看只允许确认关闭，避免判定完成后修改某一方棋子，导致界面显示结果
  与内部判定结果不一致。

## 五、识别算法鲁棒性重构

### 5.1 原算法的主要痛点

开发板实测中，原算法容易受到以下因素影响：

- 拍摄背景需要接近纯黑色。
- 光照变亮或变暗后，固定二值化阈值失效。
- 棋子没有放在预期位置时，检测框不稳定。
- 红色字符容易发生高光反射，颜色变浅甚至接近白色。
- 字符存在轻微旋转、缩放或偏移时，逐像素模板匹配得分下降。
- 单一字符提取结果错误后，后续模板匹配没有其他候选可以补救。

因此，本次没有继续依赖单一阈值，而是将识别流程改为“多候选生成 +
多特征评分”。

### 5.2 光照归一化

预处理阶段增加局部光照估计和归一化：

1. 从灰度图中估计缓慢变化的背景亮度。
2. 将原图与局部亮度进行归一化，削弱阴影和亮度渐变。
3. 使用 CLAHE 增强局部对比度。

该处理的目标是让暗处和亮处的同一字符获得更接近的灰度结构，而不是直接
根据整张图片的平均亮度选择一个固定阈值。

### 5.3 棋子检测

棋子检测不再只依赖“黑色背景上的亮棋子”，而是同时生成多组候选：

- 原始灰度候选。
- 光照归一化灰度候选。
- 亮目标二值候选。
- 暗目标二值候选。
- 局部自适应阈值候选。
- 边缘轮廓候选。

随后根据面积、长宽比、矩形程度、位置和轮廓质量对候选统一评分，选出最像
棋子的区域。这样可以降低对纯黑背景和固定摆放位置的依赖。

### 5.4 多路字符提取

同一个棋子 ROI 不再只生成一张字符二值图，而是生成多种候选：

- 局部自适应阈值候选。
- 黑帽变换得到的暗色笔画候选。
- 红色色差得到的红色笔画候选。
- 多种掩膜组合后的候选。

每个候选都会经过轮廓筛选、字符区域合并、边缘留白和尺寸归一化。模板识别器
对所有候选分别评分，最终选取整体得分最高的结果。

这一改动是字符准确率明显提升的主要原因：即使某一路受到反光或背景影响，
其他提取方式仍可能保留正确笔画。

### 5.5 字符结构修复

旧提取过程可能将封闭字符区域整体填白，导致“团”等字符内部笔画丢失。
本次调整轮廓处理方式，尽量保留：

- 字符内部空洞。
- 相邻但不连通的笔画。
- 较细的横竖笔画。
- 因二值化造成的局部断裂结构。

### 5.6 模板匹配

现有模板库没有修改。识别阶段从单一像素相关性扩展为多指标联合评分：

- 笔画重叠程度。
- 查询字符到模板轮廓的距离。
- 模板轮廓到查询字符的距离。
- 归一化相关性。
- 小范围旋转和尺度变化下的最高得分。

双向轮廓距离可以避免只从一个方向计算时出现“少一部分笔画仍然得分很高”
的问题。小范围旋转和尺度搜索则降低摆放角度与字符大小变化的影响。

模板的旋转版本、缩放版本和距离图会被缓存，避免 X210 每次识别都重复进行
相同计算。

## 六、红色棋子专项优化

### 6.1 红棋误判原因

红色棋子比黑色棋子更容易受光线影响，主要原因包括：

- 红色油墨表面反光后，局部区域亮度升高、饱和度下降。
- 强高光区域可能接近白色，固定 HSV 红色阈值无法覆盖。
- 暗光下红色的 RGB 数值整体降低，可能被误认为黑色。
- 摄像头自动白平衡会改变红、绿、蓝通道之间的比例。
- 背景或棋子底色可能带有轻微暖色，单纯判断红通道容易产生误报。

### 6.2 相对红度判断

颜色判断不再只使用固定 HSV 范围，而是先估计当前棋子自身的中性底色，再判断
笔画像素相对于底色是否明显偏红。

使用的颜色描述包括：

- RGB：`R - max(G, B)`。
- Lab：`a*` 通道相对于棋子中位数的偏移。
- YCrCb：`Cr` 通道相对于棋子中位数的偏移。

像素需要至少满足三种颜色描述中的两种，才会形成红色种子。这种方式既可以
覆盖低饱和红色，也能减少暖色背景噪声造成的误判。

### 6.3 高光恢复

高光区域本身可能已经没有明显红色，因此不能直接通过颜色阈值识别。
本次采用邻域恢复方式：

1. 先找到可信的红色笔画种子。
2. 对红色种子进行适度扩张。
3. 检测亮度高、饱和度低的高光像素。
4. 只有当高光像素位于红色种子的邻域中时，才将其恢复为红色笔画。

这样既能补回红字内部的白色反光，又不会把棋子底色或背景中的普通白色区域
全部误认为红色。

### 6.4 黑色判定约束

黑色判断除亮度和饱和度条件外，增加“低红色度”限制。暗红色像素即使亮度较低，
只要仍具有明显红色偏移，就不会直接归入黑色。

### 6.5 连通区域过滤

背景中的少量红色噪点不会直接决定整枚棋子的颜色。程序会检查红色掩膜中的
连通区域面积和连续性，只有形成有效笔画区域时才作为红棋证据。

## 八、今日遇到的问题及解决方式

### 8.1 纯黑背景依赖

**问题：** 原检测器使用单一亮暗关系定位棋子，背景变化后容易找错区域。

**解决：** 同时生成原始灰度、归一化灰度、双极性阈值、局部阈值和边缘候选，
再通过统一几何评分选择棋子。

### 8.2 光照导致字符二值图不稳定

**问题：** 固定阈值在暗光、过曝和渐变光照下无法保持字符完整。

**解决：** 增加局部光照归一化，并对同一 ROI 生成多种字符候选，避免单一路径
失败后直接产生最终误判。

### 8.3 红色高光被当成白色

**问题：** 红字反光区域饱和度下降，固定红色阈值会丢失这些笔画。

**解决：** 先检测可信红色邻域，再恢复被红色区域包围的高光像素。

### 8.4 暗红色被当成黑色

**问题：** 暗光下红色亮度降低，仅使用亮度判断会把暗红归类为黑色。

**解决：** 黑色掩膜增加低红色度限制，同时使用 RGB、Lab 和 YCrCb 判断相对红度。

### 8.5 字符旋转或偏移导致模板得分下降

**问题：** 严格逐像素相关性对位置、旋转和尺度变化敏感。

**解决：** 联合笔画重叠、双向轮廓距离和相关性，并在小角度、小尺度范围内搜索。

### 8.6 识别结果隐藏后无法复核

**问题：** 原状态区只能显示“已隐藏”，用户无法再次确认自己刚才选择的棋子。

**解决：** 将状态区改为可点击按钮，保存结果存在时允许打开只读弹窗查看。

### 8.7 开发板弹窗与背景区分不明显

**问题：** linuxfb 环境下弹窗与主界面颜色接近，边界不明显。

**解决：** 为所有关键弹窗增加不透明背景、实线边框和大尺寸按钮。

### 8.8 流水线偶尔出现负耗时

**问题：** 使用系统时间计时可能受到系统时钟调整影响。

**解决：** 将耗时统计改为 `std::chrono::steady_clock`。

### 8.9 部署包容易取错或忘记复制

**问题：** 手工寻找部署包可能拿到旧版本，且 Windows 与 WSL 路径容易混淆。

**解决：** `deploy/package.sh` 和 `deploy/pack.sh` 打包成功后自动复制部署包到
Windows 桌面，并通过 SHA256 检查仓库包与桌面包是否一致。

## 十、后续工作建议

1. 在 X210 上分别测试红、黑棋子的 12 个类别。
2. 每个类别至少测试正常光、强光、暗光、偏移和轻微旋转。
3. 使用以下方式保存真实误判截图：

   ```sh
   JUNQI_SAVE_LAST_CAPTURE=1 sh /opt/junqi/run.sh
   ```

4. 将 `/root/junqi_last_capture.jpg` 与实际棋名、算法输出一起记录。
5. 优先分析军长、排长、炸弹和军旗的混淆样本。
6. 根据真实样本分别判断问题属于检测、字符提取、颜色分类还是模板评分。
7. 完成一轮真实样本回归后，为当前版本创建 Git 提交和稳定版本标签。

## 十一、阶段性结论

本轮开发已经将项目从“依赖固定背景、一次识别两枚棋子”的实验型流程，推进为
更适合 X210 实际操作的独立识别与确认流程。

算法层面的主要进步是：

- 不再强依赖纯黑背景。
- 对光照、位置和小角度变化具有更强容错能力。
- 在不修改模板库的情况下显著提升字符识别表现。
- 针对红色低饱和、暗光和反光问题增加专项处理。

交互层面的主要进步是：

- 双方结果分别确认并隐藏。
- 支持人工纠错。
- 支持隐藏后再次查看。
- 双方确认后自动判定胜负。
- 界面和弹窗适配 X210 屏幕与触摸操作。
- 部署包可以自动生成并复制到 Windows 桌面。

当前版本已经通过桌面测试和 ARM 交叉编译。下一阶段最重要的工作不是继续堆叠
算法规则，而是收集真实开发板误判样本，用可复现的数据进行针对性优化。
