### 2.2 文件逐个说明

#### 2.2.1 `src/qt_main.cpp` — Qt 应用入口

```cpp
// 关键设计决策：
// 1. QFontDatabase::addApplicationFont() — Qt 5.6 兼容的字体加载方式
//    QT_QPA_FONTDIR 是 Qt 5.7+ 才支持，5.6 必须在代码中显式加载
//    搜索路径优先级: /root/junqi/fonts/ → ./fonts/ → fonts/
//
// 2. window.setGeometry(50, 50, 800, 480) — 不使用 showFullScreen()
//    在 linuxfb 上 showFullScreen() 可能触发 ioctl 模式切换导致段错误
//
// 3. QCommandLineParser — 解析 --template-dir 和 --camera 参数
//    兼容 CLI 工具的命令行接口风格
```

**核心流程**：
```
main()
 → QApplication app(argc, argv)
 → qRegisterMetaType<RecognitionOutput, BattleResult>()
 → QFontDatabase::addApplicationFont(阿里普惠体 TTF)
 → app.setFont(默认字体)
 → QCommandLineParser::process()
 → MainWindow window(templateDir, cameraDevice)
 → window.setGeometry(50, 50, 800, 480)
 → window.show()
 → app.exec()
```

#### 2.2.2 `include/junqi/main_window.h` + `src/main_window.cpp` — 主窗口

**头文件定义**（[main_window.h](../../include/junqi/main_window.h)）：
```cpp
class MainWindow : public QMainWindow {
    Q_OBJECT  // ← CMake AUTOMOC 必须扫描到此宏

    // 信号槽（主线程执行）：
    // captureBtn.clicked → onCaptureClicked() → invokeMethod(worker, "process")
    // worker.resultReady  → onResultReady()    → 更新 QLabel 文字
    // worker.errorOccurred → onError()         → 显示错误信息
    // worker.statusChanged → onStatusChanged() → 更新状态栏

    // 7 个 UI 控件：
    QLabel *titleLabel_, *leftPieceLabel_, *rightPieceLabel_,
           *battleResultLabel_, *elapsedLabel_, *statusLabel_;
    QPushButton *captureBtn_;

    // 工作线程（不在主线程中执行）：
    QThread *workerThread_;
    junqi::ProcessingWorker *worker_;  // moveToThread(workerThread_)
};
```

**当前 UI 状态（极简版，无 CSS/无自定义字体/无 Unicode 符号）**：
- `setupUi()`：纯 QVBoxLayout 布局，`resize(1024, 600)`，无 setStyleSheet、无 setMargins、无 setSpacing
- `setupWorker()`：创建 QThread + ProcessingWorker，连接信号槽，`workerThread_->start()`
- `onCaptureClicked()`：按钮 disable（防抖），`invokeMethod(worker_, "process", QueuedConnection)`
- `onResultReady()`：更新各 QLabel 文字，按钮 re-enable
- `battleText()`：返回纯中文 — "红方获胜"/"黑方获胜"/"平局（同归于尽）"/"判定无效"

#### 2.2.3 `include/junqi/processing_worker.h` + `src/processing_worker.cpp` — 工作线程

```cpp
namespace junqi {
class ProcessingWorker : public QObject {
    Q_OBJECT

    // 核心 slot（在工作线程中执行）：
    void process() {
        // 1. CameraCapture::capture(frame)     — V4L2 直驱拍照
        // 2. pipeline_->process(frame)         — 识别流水线
        // 3. judge_.judge(left, right)         — 判定胜负
        // 4. emit resultReady(output, result)  — 返回主线程
    }

    // 持有：
    CameraCapture camera_;                     // V4L2 摄像头
    Pipeline::Config pipelineConfig_;           // 配置（模板目录、置信度阈值）
    std::unique_ptr<Pipeline> pipeline_;        // 识别流水线
    BattleJudge judge_;                         // 胜负判定
};
}
```

**关键设计点**：
- 使用 `QObject::moveToThread()` 模式，而非继承 QThread
- 信号槽跨线程自动使用 `Qt::QueuedConnection`
- 自定义类型通过 `Q_DECLARE_METATYPE` + `qRegisterMetaType()` 注册
- C++14 `std::make_unique` → C++11 `std::unique_ptr<T>(new T(...))`（GCC 4.8 兼容）

#### 2.2.4 `include/junqi/camera_capture.h` + `src/camera_capture.cpp` — V4L2 摄像头直驱

**为什么不用 OpenCV VideoCapture**：
ARM 预编译的 OpenCV 3.4.16 只有 core/imgproc/imgcodecs 三个模块，**不含 videoio**。因此直接使用 Linux V4L2 内核 API。

**V4L2 采集流程**：
```
open("/dev/video0") → ioctl(VIDIOC_QUERYCAP)
 → ioctl(VIDIOC_S_FMT) ← 设置 YUYV 640×480
 → ioctl(VIDIOC_REQBUFS) ← 申请 4 个缓冲区
 → mmap() × 4 ← 内存映射，零拷贝
 → ioctl(VIDIOC_QBUF) × 4 ← 缓冲区入队
 → ioctl(VIDIOC_STREAMON) ← 开始采集
 → ioctl(VIDIOC_DQBUF) × 5 ← 丢前 4 帧（曝光稳定），取第 5 帧
 → YUYV → BGR 转换（cv::cvtColor, COLOR_YUV2BGR_YUYV）
 → 关闭: STREAMOFF → munmap → close
```

**头文件接口**（[camera_capture.h](../../include/junqi/camera_capture.h)）：
```cpp
class CameraCapture {
    bool open(const char* device_path, int width = 640, int height = 480);
    bool capture(cv::Mat& frame);  // 返回 BGR 格式 cv::Mat
    void close();
    bool isOpened() const;
};
```

#### 2.2.5 `include/junqi/battle_judge.h` + `src/battle_judge.cpp` — 对战判定

纯逻辑模块，无外部依赖（不依赖 OpenCV 也不依赖 Qt）。

**判定规则表**：

| 条件 | 结果 | 说明 |
|------|------|------|
| 任一 `character_id == -1` | `INVALID` | 识别失败 |
| 任一方为炸弹(11) | `DRAW` | 同归于尽 |
| 双方棋子相同 | `DRAW` | |
| 左为地雷(10)，右为工兵(9) | `RIGHT_WINS` | 工兵挖雷 |
| 左为地雷(10)，右非工兵 | `LEFT_WINS` | 地雷吃子 |
| 右为地雷(10)，左为工兵(9) | `LEFT_WINS` | 工兵挖雷 |
| 右为地雷(10)，左非工兵 | `RIGHT_WINS` | 地雷吃子 |
| 任一方为军旗(12) | 对方胜 | 军旗最弱 |
| 普通比大小: lid < rid | `LEFT_WINS` | ID 小的职位大 |

**棋子 ID 对照**：
```
 1=司令  2=军长  3=师长  4=旅长  5=团长  6=营长
 7=连长  8=排长  9=工兵 10=地雷 11=炸弹 12=军旗
```

#### 2.2.6 `include/junqi/qt_utils.h` — QImage ↔ cv::Mat 互转

两个 inline 函数，提供 Qt 图像和 OpenCV 矩阵之间的桥接：
- `qimage_to_mat()`：QImage → cv::Mat（BGRA 4 通道）
- `mat_to_qimage()`：cv::Mat → QImage（深拷贝，线程安全，支持 CV_8UC3 和 CV_8U）

#### 2.2.7 `include/junqi/result.h` — 核心数据结构

```cpp
enum class PieceColor { RED, BLACK, UNKNOWN };
enum class BattleResult { LEFT_WINS, RIGHT_WINS, DRAW, INVALID };

struct PieceResult {
    PieceColor color;        // 颜色分类结果
    std::string character;   // 文字识别结果："司令"/"军长"/...
    int character_id;        // 1-12，-1 表示失败
    float confidence;        // 0.0 ~ 1.0
    cv::Rect bounding_box;   // 棋子边框
};

struct RecognitionOutput {
    PieceResult left_piece;  // 左棋（默认红方）
    PieceResult right_piece; // 右棋（默认黑方）
    bool success;            // 两枚都识别成功
    std::string error_message;
    double elapsed_ms;       // 处理耗时
};
```

### 2.3 识别流水线（libjunqi_core.a）内部模块

```
Pipeline::process(frame)
  │
  ├─ 1. Preprocessor::process(frame)
  │     └→ bilateralFilter 去噪 + CLAHE 对比度增强 → gray + color
  │
  ├─ 2. Detector::detect(gray, color)
  │     └→ Otsu/自适应二值化 + 轮廓分析 + 几何约束 → left & right ROIs
  │
  ├─ 3. 对每枚棋子:
  │     ├─ ColorClassifier::classify(piece_color)
  │     │   └→ HSV + Lab + YCrCb 三空间投票 → RED / BLACK
  │     ├─ CharacterExtractor::extract(piece_gray)
  │     │   └→ 笔画轮廓合并 + padding + normalize(64×64)
  │     └─ Recognizer::recognize(char_binary)
  │         └→ TM_CCOEFF_NORMED (±5° ±10%) + Hu 矩回退
  │
  └─ 4. 组装 → RecognitionOutput
```

## 6. 4 轮调试迭代记录

### 迭代 1：tslib + 字体 + 窗口尺寸修复

**初始症状**：
- `ts_config() failed` — 缺少 `/etc/ts.conf`
- `QFontDatabase: Cannot find font directory /home/kt/app/qt5.6.2/lib/fonts` — 编译机硬编码路径
- `window.show()` → Segmentation fault

**修改**：
- `deploy/run.sh`：添加 tslib 环境变量，自动生成 `ts.conf`
- `deploy/run.sh`：部署阿里普惠体 TTF 到 `/root/junqi/fonts/`
- `src/qt_main.cpp`：`show()` → `showFullScreen()`（linuxfb 推荐用法）
- `src/main_window.cpp`：`resize(480, 640)` → `resize(1024, 600)`（匹配 LCD）

**结果**：tslib 错误消失，字体警告消失，**仍然 Segfault**。

---

### 迭代 2：Qt 5.6 字体加载方式修复

**分析**：`QT_QPA_FONTDIR` 是 Qt 5.7+ 才支持的环境变量，开发板 Qt 5.6.2 不认。

**修改**：
- `src/qt_main.cpp`：改用 `QFontDatabase::addApplicationFont()` 显式加载 TTF
- 搜索路径优先级：`/root/junqi/fonts/` → `./fonts/` → `fonts/`
- 加载成功后 `app.setFont()` 设为应用默认字体

**结果**：
```
[FONT] Loaded: "/root/junqi/fonts/AlibabaPuHuiTi-3-55-Regular.ttf" id: 0
[FONT] Families: ("Alibaba PuHuiTi 3.0")
[FONT] Set default font to: "Alibaba PuHuiTi 3.0"
```
字体加载成功，**showFullScreen() 仍然 Segfault**。

---

### 迭代 3：去掉所有 CSS/QSS 样式

**分析**：`setStyleSheet()` 中的 `border-radius`、渐变等属性可能在 linuxfb 平台触发未定义行为。

**修改**：
- `src/main_window.cpp`：删除全部 `setStyleSheet()` 调用（静态 + 动态着色）
- `src/main_window.cpp`：删除全部单独 `QFont` 设置（完全依赖 `app.setFont()`）
- `src/main_window.cpp`：删除 `QFrame::HLine` 分隔线
- `src/main_window.cpp`：`★`/`●` Unicode 符号 → 纯中文

**结果**：**仍然 Segfault**，确认 CSS/StyleSheet 不是崩溃原因。

---

### 迭代 4：最小化 UI + show() 替代 showFullScreen() + 诊断

**分析**：
- `showFullScreen()` 可能触发 framebuffer 模式切换 ioctl，S5PV210 内核不支持
- 窗口尺寸接近 framebuffer 边界可能越界

**修改**：
- `src/qt_main.cpp`：`showFullScreen()` → `show()` + `setGeometry(50, 50, 800, 480)`
- `src/main_window.cpp`：删除 `setContentsMargins()`、`setSpacing()`、`setMinimumHeight()`
- `deploy/run.sh`：添加 `QT_QPA_FB_FORCE_FULLSCREEN=0`
- `deploy/run.sh`：添加完整诊断输出（fbset, ldd, free, ls plugins）
- `deploy/pack.sh`：新建一键构建+打包脚本

**结果**：**仍然 Segfault**，但获得关键诊断信息：

```
=== DIAG: framebuffer ===
1024,1200           ← 虚拟分辨率是 1024×1200，双倍高度
=== DIAG: free memory ===
Mem: 320M total, 207M free
=== DIAG: Qt plugins ===
libqlinuxfb.so  libqminimal.so  libqoffscreen.so
```

## 7. 诊断发现与排除表

### 7.1 已排除的因素

| 因素 | 排除方式 | 结论 |
|------|---------|:--:|
| 中文字体缺失 | `QFontDatabase::addApplicationFont()` 确认加载成功 | ❌ |
| CSS/StyleSheet | 全部删除仍崩溃 | ❌ |
| 窗口尺寸越界 | 800×480 远小于 1024×600 | ❌ |
| 内存不足 | `free -m` 确认 207MB 空闲 | ❌ |
| OpenGL 依赖 | `readelf -d junqi_gui` 无 GL NEEDED 条目 | ❌ |
| `showFullScreen()` vs `show()` | 两者均崩溃 | ❌ |
| OpenCV 动态库缺失 | `LD_LIBRARY_PATH` 设置正确，`readelf` 解析正常 | ❌ |

### 7.2 重要发现

| 发现 | 详情 |
|------|------|
| **framebuffer 虚拟分辨率异常** | 物理 1024×600，但 `/sys/class/graphics/fb0/virtual_size` = `1024,1200`（双倍高度） |
| **offscreen 平台可用** | `libqoffscreen.so` 存在于板子 Qt 插件目录中 |
| **linuxfb 是唯一显示插件** | 板上无 eglfs/wayland/xcb 插件 |

### 7.3 崩溃时序推断

```
main()
 → QApplication app(...)          ← OK
 → addApplicationFont()           ← OK（[FONT] Loaded 出现在日志中）
 → MainWindow ctor: setupUi()     ← 大概率 OK（纯 QLabel/QPushButton 构造，无 fb 交互）
 → MainWindow ctor: setupWorker() ← 大概率 OK（纯 QThread 创建 + connect）
 → window.setGeometry(50,50,800,480) ← 不一定 OK（QWidget 几何变更可能触发平台调用）
 → window.show()                  ← 崩溃点
                                    linuxfb 写 /dev/fb0 时 ioctl 返回异常
                                    或 kb 驱动对双倍高度 fb 处理出错
```
